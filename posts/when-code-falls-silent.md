我曾以为优秀的工程师是那种能写出最复杂代码的人。后来我在一份十年前的代码库里待了三个月——那是一段几乎没什么注释、却**什么都不必解释**的代码。它沉默地工作着，像一堵好墙。那一刻我才意识到：*克制是一种工程能力，而非审美偏好*。

这篇文章里，我想谈谈那些被刻意留下的空白——函数之间的负空间、被删掉的抽象层、以及一句「这里其实不需要它」背后所节省的未来三周。它们不会出现在 commit message 的醒目位置，却悄悄决定了系统会不会在第三年崩塌。

## 一、沉默的墙

那是一份支付对账系统。作者早已离职，文档只有半页 README，写着「每天凌晨跑一次，失败会发邮件」。我接手它是因为要对账逻辑做一处小改动。

我做好了在丛林里开路的准备：注释稀薄、命名晦涩、到处是「不要动这里」的玄学注释。但我打开第一个文件，看到的却是这样的东西——

```js
// 把当日订单按渠道分组，对账后落库
const byChannel = group(orders, (o) => o.channel);
for (const [channel, list] of Object.entries(byChannel)) {
  const ledger = reconcile(list, channelRates[channel]);
  await store(ledger);
}
```

没有多余的抽象，没有 `ReconciliationServiceFactory`，没有为了「将来可能要支持新渠道」而预留的插件系统。它只做今天要做的事，把它做好。

## 二、负空间

好的代码像好的排版——它的美不在写了什么，而在**没写什么**。那半行留白，那个被省略的参数对象，那个本可以抽成函数却留在原处的循环——它们是工程师留给未来读者的呼吸空间。

我们这一代人被教育「DRY」「抽象」「复用」，于是我们把三行代码提炼成一个类，又把类提炼成框架。但没人告诉我们：每一次抽象都是一次**对未来读者的承诺**，承诺这个抽象会在所有它声称覆盖的场景里都正确工作。承诺越多，违约的可能越大。

## 三、删掉，是一种创作

我最终对那份对账系统做的改动是：删掉了一个 2019 年加的「多币种预处理」层——它已经不需要了，但没人敢删。删掉之后，主流程从 47 行变成 31 行，测试照常绿。

那 16 行的消失，是我那一年最得意的提交。

> 克制不是不做，而是知道什么**不必做**。
